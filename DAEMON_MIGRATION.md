# Переход на vpnctld — статус и что осталось

Этот документ отслеживает переход vpnctl с модели "каждый вызов CLI/TUI — самостоятельный процесс, координирующийся через `active.json` на диске" на модель "привилегированный демон `vpnctld` единолично владеет netns/iptables/engine-состоянием в памяти, `vpnctl` — тонкий клиент". Мотивация и общий план — в роадмап-ТЗ ("Часть B"). Здесь — конкретно что сделано и что нет, по состоянию кода.

## Модель демона

Единый системный root-демон на всю машину (не per-user) — `vpnctld`, слушает Unix-сокет `/run/vpnctl.sock` (переопределяется через `$VPNCTL_SOCKET` у клиента, `--socket` у демона). Соответствует текущей реальности: namespace `vpnctl0` — фиксированное имя, активен всегда только один профиль системно.

Протокол (`internal/rpc`): `[4 байта big-endian длина][JSON]`, один запрос — одно соединение (без мультиплексирования), конверт с полем `api_version` (несовпадение — явная ошибка, не крэш). Одно исключение — `Exec` (см. ниже): после начального хендшейка то же соединение переходит в потоковый режим кадров `[1 байт type][4 байта длина][payload]` (`internal/rpc/stream.go`).

## Сделано

### Фаза 1 — фундамент + простые команды

- **`internal/rpc`** — фрейминг, конверт запрос/ответ, методы `Ping/Activate/Deactivate/Status/TestConnectivity/ListProcesses/KillProcess`.
- **`internal/vpnctld`** — сервер: in-memory состояние под мьютексом (замена `internal/netguard`'s file-backed `ActiveState` — см. ниже про то, что из Части A теперь неактуально), держит один `netguard.NewLinuxEngine(false)`, health-check как горутина с `context.CancelFunc` вместо detached-процесса.
- **`cmd/vpnctld`** — entrypoint демона: требует root, слушает сокет (подчищает stale-сокет от предыдущего аварийно завершённого инстанса), грациозный `Deactivate`+`Teardown` на SIGTERM/SIGINT.
- **`internal/vpnctlclient`** — тонкий клиент: резолвит профиль локально (`profile.Find`, демон никогда не читает `~/.config/vpnctl/profiles` ни для одного пользователя), шлёт уже разрешённый профиль по сокету.
- **Конвертированы**: `vpnctl use/down/status/test` — полностью через демон, root/sudo не нужен.
- **`vpnctld`'s own state dir**: `netguard.StateDir()` резолвит через "домашнюю директорию реального пользователя" — осмысленно для старой per-invocation CLI-модели, но у демона нет "вызывающего пользователя" вообще, а `systemd-run`/`.service` не даёт `$HOME` — падало с `$HOME is not defined` (поймано живьём на стенде). Починено через `VPNCTL_STATE_HOME=/var/lib/vpnctld` (флаг `--state-dir`), выставляемый демоном себе самому при старте — переиспользует ту же переменную, что уже вводили в Части A для `prerm`. Итоговый путь получается `/var/lib/vpnctld/.local/state/vpnctl/...` (наследие того, что `StateDir()` всегда дописывает `.local/state/vpnctl` к переданному "home") — работает корректно, но при packaging стоит решить, нужен ли более прямой путь вида `/var/lib/vpnctl/...` без вложенности.

### Фаза 2 — `vpnctl run` за демоном (PTY/pipe-прокси)

- **Новые зависимости**: `github.com/creack/pty` (сервер — реальный PTY), `golang.org/x/term` (клиент — raw mode + размер терминала).
- **`internal/rpc/stream.go`** — `Exec`-метод: хендшейк как у обычного RPC (`ExecParams{Mode, Argv, Env, DropUID, DropGID, Cols, Rows}` → `ExecStartedResult{PID}`), дальше то же соединение несёт кадры `FrameStdin/FrameStdout/FrameStderr/FrameResize/FrameExit`.
- **`internal/vpnctld/exec.go`** — три режима: `cli` (пайпы, без PTY — как и раньше у `vpnctl run` без флагов, реальный терминал никогда явно не аллоцировался, просто наследовался напрямую), `tui` (настоящий PTY через `creack/pty`, `FrameResize` → `pty.Setsize`), `gui` (detached, `/dev/null`, демон реапит и untrack'ает процесс сам — в отличие от старого `internal/run.GUI`, который никогда не мог этого сделать, потому что процесс vpnctl завершался сразу же). Обрыв соединения ДО завершения процесса → демон убивает его (SIGTERM, потом SIGKILL) — это и есть "отмена", отдельного управляющего кадра не нужно.
- **`ListProcesses`/`KillProcess` теперь реальные**: `Server.processes` наполняется `Exec`'ом, `KillProcess` шлёт сигнал напрямую (`syscall.Kill`) — демон всегда root, поэтому проще клиентской EPERM-эквилибристики из `internal/actions/processes.go`. `deactivateLocked()` перед `Teardown()` убивает все трекнутые процессы (аналог `netguard.killTrackedProcesses`, только по in-memory списку, а не по `active.json`, который демон не пишет) — без этого `vpnctl down`/остановка демона осиротили бы `run`-сессии внутри уничтожаемого namespace.
- **Найденная и починенная гонка**: `ptmx.Close()` в основной горутине `execPTY` конкурировал с `pty.Setsize`/`Write` в горутине, читающей входящие кадры (`relayPTYInput`) — обе трогают один и тот же fd PTY. Поймано `go test -race`, а не угадано. Фикс: перед закрытием `ptmx` основная горутина форсирует выход `relayPTYInput`, дёргая `conn.SetReadDeadline(time.Now())` и дожидаясь её завершения — только потом закрывает `ptmx`.
- **Конвертирован**: `vpnctl run` (все три режима — без флага, `--tui`, `--gui`) — root/sudo не нужен.

### Фаза 3 — перевод `internal/tui` на клиента демона (последний непереведённый кусок)

- **Простые RPC** (`internal/tui/model.go`, `psview.go`): `refreshStatusCmd`/`activateCmd`/`Deactivate`/`loadProcessesCmd`/`killSelectedProcessCmd`/`testCmd` — все переведены на `vpnctlclient.*` напрямую, сигнатуры сообщений (`statusMsg`/`activateDoneMsg`/`testDoneMsg`) сохранены, только `testDoneMsg` теперь несёт `rpc.TestConnectivityResult` (готовый `.Output` вместо живого стриминга).
- **Логи** (`mainview.go`): новый RPC `GetLogTail(lines int) (text string)` — у демона нет `active.json`/`EngineLog`-пути, читаемого с клиента напрямую (state dir демона root-only). Хендлер переиспользует уже существующий `tailLogFile`. Результат кэшируется в `Model.logTail`, обновляется на том же 2-секундном tick'е, что и статус/процессы — не блокирующий RPC на каждый рендер.
- **Запуск программ через демон** (`runview.go`, `appsview.go`): новый адаптер `internal/tui/execcmd.go` (`daemonExecCommand`) реализует интерфейс `tea.ExecCommand` (`Run()`/`SetStdin`/`SetStdout`/`SetStderr` — последние три no-op, `vpnctlclient.Exec`'s `relay()` уже пишет напрямую в реальные os.Stdin/Stdout/Stderr), передаётся в `tea.Exec` вместо `tea.ExecProcess(*exec.Cmd, ...)`. `runview.go`'s Run-экран и `appsview.go`'s CLI/TUI-типа apps используют этот адаптер; GUI-типа apps зовут `vpnctlclient.Exec(rpc.ExecModeGUI, ...)` напрямую (fire-and-forget, детач), `sysuser.ResolveGUIEnv` строит env.
- **`cmd/vpnctl/tui.go`**: `actions.RequireRoot()` убран — весь `internal/tui` общается с сетью только через сокет демона, бинарь `vpnctl` больше нигде не требует root.
- **Уборка**: `internal/actions` удалён целиком (Activate/Deactivate/CurrentStatus/TestConnectivity/RequireRoot/ListProcesses/KillProcess/spawnHealthCheckDaemon — все вызовы теперь идут через `vpnctlclient`). `internal/healthcheck` урезан до `Interval()`/`DefaultInterval` (`Run()` и file-backed loop удалены как мёртвый код — обслуживал только старую detached-daemon модель; `internal/vpnctld/healthcheck.go`'s `healthCheckLoop` — единственный потребитель пакета теперь, использует только `Interval()`). Скрытый `actions.HealthCheckDaemonArg` dispatch-case убран из `cmd/vpnctl/main.go`. `internal/run` удалён целиком — `ResolveGUIEnv`+хелперы (`guiEnvKeys`/`scanProcEnvForUser`) переехали в `internal/sysuser` (`guienv_linux.go`/`guienv_other.go`), `CLI`/`GUI`/`foreground`/`exitCodeOf`/`unwrapExitError`/`Type*` удалены как мёртвый код (оба потребителя, `cmd/vpnctl/run.go` и `internal/tui/appsview.go`, уже звали `vpnctlclient.Exec` напрямую).
- **Намеренно НЕ тронуто**: `internal/netguard/state.go`'s file-backed `ActiveState`/`WriteActiveState`/`LoadActiveState`/flock-механизм — не мёртвый код, `LinuxEngine.Setup/Teardown/Status/ensureIPForward/restoreIPForward` (реальная имплементация, которой `vpnctld` пользуется) сама вызывает эти функции для собственных внутренних нужд, независимо от того, кто сверху. Flock избыточен в демон-модели (гонок между процессами больше нет), но переписывание `linux.go` под чисто in-memory — отдельная самостоятельная задача, не часть перевода TUI.
- **Живая проверка на стенде** (172.16.20.36): собраны свежие `vpnctl`/`vpnctld`, задеплоены, демон запущен вручную (`setsid vpnctld`, group `vpnctl` + сокет `root:vpnctl 0660` — временная замена будущего `postinst`, см. ниже). Подтверждено через `tmux`: голый `vpnctl` стартует без sudo, видит профиль, активированный отдельно через `vpnctl use` (демон — общий источник правды теперь и для TUI, и для CLI, старое ограничение "не смешивать TUI и остальные команды" снято); активация/деактивация/test/логи через TUI работают; GUI-app (Firefox) запускается детач и отслеживается в `ps`/убивается через `x`; CLI-app (curl) делает полный terminal takeover и корректно возвращает управление TUI; TUI-app (htop) получает настоящий PTY, полностью интерактивен (F-клавиши, полноэкранная перерисовка), выход чистый. После теста: `vpnctld` остановлен по SIGTERM, own teardown прошёл чисто (netns/iptables-правила не осиротели).

### Фаза 4 — systemd + packaging

- **`packaging/vpnctld.service`** (новый файл): `Type=notify`, `Restart=on-failure`, **`KillMode=process`** — критично, задокументировано прямо в юните: дефолтный `KillMode=control-group` при `systemctl stop` шлёт SIGTERM всей cgroup, включая дочерний процесс, который демон порождает для graceful teardown (`awg-quick down` через nsenter, внутри `Shutdown()`→`deactivateLocked()`→`handle.Stop()`), убивая его на середине и оставляя namespace висеть — см. `## Errors and fixes`-класс проблем, пойманных живьём на стенде в Части 2 (`vpnctl run`). `ExecStart=/usr/local/bin/vpnctld` (без `-socket-group` — дефолт флага уже `vpnctl`).
- **`cmd/vpnctld/main.go`**: добавлен флаг `-socket-group` (дефолт `vpnctl`) — демон сам делает `chown root:<group> 0660` на сокет сразу после `net.Listen`, а не постфактум через `postinst` (убирает гонку "сокет появился, но ещё не перехвачен" между стартом демона и первым клиентским подключением). Также добавлен минимальный `sd_notify` (сырой datagram-write в `$NOTIFY_SOCKET`, без внешней зависимости на `go-systemd`) — `READY=1` сразу после успешного `Listen`, `STOPPING=1` перед `Shutdown()`, что и требует `Type=notify` в юните.
- **`postinst`**: `setup_daemon_group()` — `groupadd -f vpnctl` + `usermod -aG vpnctl $SUDO_USER` (если запускающий пользователь не root). `migrate_from_file_based_model()` — та же PID-kill/namespace-removal логика, что в `prerm`'s `force_teardown`, прогоняется по `/root/.local/state/vpnctl` и всем `/home/*/.local/state/vpnctl` **до** первого `systemctl enable --now vpnctld` — иначе апгрейд с файловой версии на демон-версию оставил бы осиротевший `vpnctl0`/процессы от старой модели, о которых демон ничего не знает. `start_daemon()` — `systemctl daemon-reload && systemctl enable --now vpnctld`, с graceful degradation (`WARNING`, не `exit 1`) если `systemctl` недоступен.
- **`prerm`**: весь старый per-user `vpnctl down`-танец (который звал уже удалённую по духу файловую модель через сам бинарь `vpnctl`) заменён на `systemctl stop vpnctld` — демон сам корректно сносит namespace/iptables/процессы через свой `Shutdown()`. `force_teardown` (то же самое PID-kill/namespace-removal, что в `postinst`'s миграции) оставлен как fallback — запускается только если `stop_daemon` не смог подтвердить чистую остановку (`systemctl` недоступен/`stop` вернул ошибку) или namespace всё ещё присутствует после. `systemctl disable vpnctld` — только при `$1 = remove` (не при `upgrade`), и до удаления файлов пакета, поскольку `disable` должен прочитать `[Install]`-секцию ещё существующего unit-файла.
- **`postrm`**: в `purge`-ветке — удаление `/lib/systemd/system/vpnctld.service` (defensive, на случай если nfpm/dpkg почему-то не удалил свой же файл) + `systemctl daemon-reload`, плюс снос `/var/lib/vpnctld` (state dir демона — sysctl backup, engine-конфиги, логи). Группа `vpnctl` намеренно **не** удаляется даже при purge — задокументировано в коде почему (риск неожиданно тронуть чужое членство в группе не стоит того).
- **`nfpm.yaml`**: добавлены `vpnctld` (`/usr/local/bin/vpnctld`, 0755) и `vpnctld.service` (`/lib/systemd/system/vpnctld.service`, 0644) в `contents:`; `systemd` добавлен в `depends:`; версия поднята `1.0.4` → `1.2.0` (синхронизация с релизным тегом); build-инструкция в шапке дополнена `go build -o packaging/vpnctld ./cmd/vpnctld`.
- **Живая проверка на стенде** (172.16.20.36), реальным `.deb` через `apt`, не вручную скопированными бинарями: `sudo apt install ./vpnctl_1.2.0_amd64.deb` поверх ранее стоявшей `1.0.15` — `postinst` нашёл настоящий (не синтетический) осиротевший `active.json` от старого тестирования, прогнал по нему миграцию и удалил его; `vpnctld.service` включился и стартовал (`systemctl status` → `active (running)`, `Type=notify`/`sd_notify` отработали — не зависло в ожидании готовности); сокет создался сразу с правами `root:vpnctl 0660` (демон сам себе chown'ит, без дополнительного шага в `postinst`). `vpnctl use switz_vpn` и `vpnctl run -- curl ifconfig.me` без sudo отработали корректно (трафик реально идёт через тоннель — вернулся IP VPN-сервера). `sudo apt purge vpnctl` **с активным профилем** (стресс-тест именно на `KillMode=process`) прошёл чисто: `vpnctld` остановился, снёс namespace/iptables/процессы через свой `Shutdown()` до того, как файлы пакета удалились; после purge — `ip netns list` пусто, `iptables -S FORWARD` без vpnctl-правил, ни `vpnctld`, ни `awg-quick`/`amneziawg-go` не осталось, unit-файл удалён (`systemctl status vpnctld` → `could not be found`), пакет полностью вычищен из `dpkg -l`. Повторный `apt install` с нуля (после purge) тоже прошёл чисто — подтверждён fresh-install путь, не только upgrade.

### Отдельная, самостоятельная задача (не блокирует остальное)

- `internal/netguard`'s file-backed `ActiveState`/flock-механизм — переписать `linux.go` под чисто in-memory состояние демона, раз межпроцессная гонка больше архитектурно невозможна. Не мёртвый код сейчас (см. выше), просто избыточен.
